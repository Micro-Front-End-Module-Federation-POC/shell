What is Micro Frontend Architecture?

Micro Frontends extend the concept of microservices to the frontend layer. Instead of one large Angular app, the UI is split into smaller, independently developed, tested, and deployed applications each owned by a separate team.

📺 Monolith

One Angular project. All features developed and deployed together. A single broken build blocks everyone.

Tight coupling
→

🪟 Micro Frontends

Multiple Angular apps. Each team ships independently. Composed at runtime in the browser.

Loose coupling

Architecture of This POC

Three separate Angular projects, each with its own build pipeline and deployment. The Shell discovers remotes at runtime via a JSON manifest — no rebuild needed when remotes change.

How It Works at Runtime

1
Shell boots

Fetches module-federation.manifest.json to discover remote URLs (environment-aware: localhost vs GitHub Pages).

2
User navigates to a remote route

Angular's router calls loadRemoteModule(). Webpack fetches the remote's remoteEntry.js.

3
Webpack negotiates shared scope

Angular, RxJS, and the shared library are loaded only once.

4
Remote component renders inside Shell

The remote's root component is injected into the Shell's <router-outlet> as if it were a local component.

5
State flows via shared singleton

Products writes to SharedStateService. Cart and Shell's navbar react instantly — same BehaviorSubject instance in memory.

!
Remote fails to load → Fallback

If remoteEntry.js is unreachable, the .catch() handler renders a local fallback component. Shell stays fully functional.

Key Benefits

🚀 Independent Deployments

Each team can release on their own cadence without coordinating with others.

🛠 Team Autonomy

Teams own their MFE end-to-end codebase, CI, deploy, and rollback.

🛡 Fault Isolation

One remote crashing does not bring down the entire application.

🔄 Incremental Upgrades

Upgrade one MFE to a newer Angular version without touching others.

⚡ Smaller Bundles per Team

Each team's build is scoped to their feature , making faster CI and focused bundle analysis.

When to Use Micro Frontends

✅ Good Fit

  • Large org with multiple independent teams
  • Different release cadences per domain
  • Need to run A/B experiments on isolated sections
  • Gradual migration from a legacy app
  • Different tech stacks per team (React + Angular)

❌ Poor Fit

  • Small team (< 5 devs) — overhead outweighs benefits
  • Tight feature coupling across all sections
  • No established CI/CD pipeline
  • Performance-critical app where bundle overhead matters
  • Short-lived prototype or MVP
