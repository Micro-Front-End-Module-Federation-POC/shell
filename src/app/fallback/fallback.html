Checkout MFE — Not Deployed

This is intentional. The mfe-checkout remote is listed in the Module Federation manifest but has never been deployed. When the shell tries to load it at runtime, the network request for remoteEntry.js fails — and the .catch() handler kicks in, rendering this component instead.

How the Fallback Works

1
Shell navigates to /demo/checkout

Angular lazy-loads the route by calling loadRemoteModule() with remoteName: 'mfe-checkout'.

2
Webpack fetches remoteEntry.js

Module Federation reads the manifest to find the remote URL, then makes a network request for the entry file.

3
Network request fails (404 / unreachable host)

The remote is not deployed, so the fetch throws an error and the Promise rejects.

4
.catch(() => Fallback) in demo.routes.ts

Instead of crashing the entire shell, the catch handler returns the local Fallback component — keeping the host app fully functional.

Why This Matters

🛡 Resilience

One MFE going down does not take the entire platform offline.

🔄 Independent Deploys

Teams can release on their own cadence the shell handles the rest.

👁 UX Continuity

Users see a meaningful message instead of a blank screen or JS error.

🔧 Dev Isolation

Other teams can develop and test locally without needing checkout running.
