Study Guide

How to Implement Module Federation

All commands, concepts, pitfalls, and interview answers — in one place.

01

Project Setup

📚 Theory

This POC is built using Angular CLI without any monorepo tool like Nx. Each application is a completely standalone Angular project with its own package.json, node_modules, and build pipeline. They are developed, versioned, and deployed independently, and integrated only at runtime via Webpack Module Federation. This approach demonstrates that you do not need a monorepo to implement Micro Frontends — Angular CLI alone is sufficient.

Create each app separately

terminal
ng new shell --routing --style=scss
ng new mfe-products --routing --style=scss
ng new mfe-cart --routing --style=scss

Add Module Federation to each app

terminal
# Shell becomes the HOST
ng add @angular-architects/module-federation --type=host --port=4200

# Remotes are configured as REMOTE
ng add @angular-architects/module-federation --type=remote --port=4201
ng add @angular-architects/module-federation --type=remote --port=4202
💡 What this does: Generates webpack.config.js, updates angular.json to use ngx-build-plus:browser instead of the default Angular builder, and creates bootstrap.ts with a dynamic import to defer Angular's bootstrap until Webpack has negotiated all shared modules.

Environment files (for manifest URL switching)

src/environments/environment.ts
export const environment = {
  production: false,
  manifestPath: '/module-federation.manifest.json'
};
src/environments/environment.prod.ts
export const environment = {
  production: true,
  manifestPath: '/shell/module-federation.manifest.prod.json'
};
angular.json — fileReplacements in production config
"fileReplacements": [
  {
    "replace": "src/environments/environment.ts",
    "with":    "src/environments/environment.prod.ts"
  }
]
02

Shared Library

📚 Theory

A shared library is a separately built and published Angular library consumed by all MFEs as an npm package. It avoids duplicating common code. When configured as singleton: true in the Webpack shared scope, only one instance is loaded at runtime enabling true cross-MFE state sharing via a single BehaviorSubject.

Create the library workspace

terminal
ng new shared-state-lib --no-create-application
cd shared-state-lib
ng generate library shared-state --prefix=mfe

Service inside the library

projects/shared-state/src/lib/shared-state.service.ts
export interface Course {
  id: number; title: string; instructor: string;
  price: number; rating: number; reviews: number;
  level: string; icon: string; tag: string;
}

@Injectable({ providedIn: 'root' })
export class SharedStateService {
  private state = new BehaviorSubject<Course[]>([]);
  state$ = this.state.asObservable();

  addCourse(course: Course) {
    if (!this.state.value.find(c => c.id === course.id))
      this.state.next([...this.state.value, course]);
  }

  removeCourse(id: number) {
    this.state.next(this.state.value.filter(c => c.id !== id));
  }
}

Build & publish

terminal
# Build
ng build shared-state

# Publish from dist output
cd dist/shared-state
npm publish --registry=https://npm.pkg.github.com

.npmrc — scope the package to GitHub Packages

.npmrc (in consuming apps)
@your-org:registry=https://npm.pkg.github.com
//npm.pkg.github.com/:_authToken=${NODE_AUTH_TOKEN}

Install in consuming apps

terminal (in shell / mfe-products / mfe-cart)
npm install @your-org/shared-state
💡 Why publish? Path aliases pointing to local source only work on your machine. Publishing to a registry means CI/CD pipelines and other developers can install the exact same built version — reproducible and versioned.
03

Module Federation Configuration

📚 Theory

Webpack Module Federation works through two roles — Host and Remote. The Remote exposes modules (components, routes). The Host consumes those modules at runtime by fetching the remote's remoteEntry.js. The Shell uses a manifest (a JSON file) — to discover remote URLs dynamically, avoiding hardcoded URLs in the build.

Shell webpack.config.js (Host)

webpack.config.js
const { shareAll, withModuleFederationPlugin } = require('@angular-architects/module-federation/webpack');

module.exports = withModuleFederationPlugin({
  shared: {
    ...shareAll({ singleton: true, strictVersion: true, requiredVersion: 'auto' }),
    '@your-org/shared-state': { singleton: true, requiredVersion: 'auto' },
  },
});

Remote webpack.config.js

webpack.config.js (mfe-products)
module.exports = withModuleFederationPlugin({
  name: 'mfe-products',
  exposes: {
    './Component': './src/app/app.ts',
  },
  shared: {
    ...shareAll({ singleton: true, strictVersion: true, requiredVersion: 'auto' }),
    '@your-org/shared-state': { singleton: true, requiredVersion: 'auto' },
  },
});

Manifest files (dev vs prod)

module-federation.manifest.json (dev)
{
  "mfe-products": "http://localhost:4201/remoteEntry.js",
  "mfe-cart":     "http://localhost:4202/remoteEntry.js"
}
module-federation.manifest.prod.json (production)
{
  "mfe-products": "https://your-org.github.io/mfe-products/remoteEntry.js",
  "mfe-cart":     "https://your-org.github.io/mfe-cart/remoteEntry.js"
}

main.ts — Load manifest before bootstrap

src/main.ts
import { loadManifest } from '@angular-architects/module-federation';
import { environment } from './environments/environment';

loadManifest(environment.manifestPath)
  .catch(err => console.error(err))
  .then(() => import('./bootstrap'))
  .catch(err => console.error(err));

Shell routing — lazy load remotes with fallback

demo.routes.ts
import { loadRemoteModule } from '@angular-architects/module-federation';
import { Fallback } from '../fallback/fallback';

{
  path: 'products',
  loadComponent: () =>
    loadRemoteModule({
      type: 'manifest',
      remoteName: 'mfe-products',
      exposedModule: './Component',
    }).then(m => m.App).catch(() => Fallback),
}
💡 Why manifest? Without a manifest, remote URLs are hardcoded in webpack.config.js at build time. The manifest approach allows URLs to be changed without rebuilding the Shell , making it environment-aware at runtime.
04

Shared State Between MFEs

📚 Theory

Cross-MFE state sharing requires a singleton service — one instance shared across all remotes. This is achieved by: (1) having all apps import from the same npm package, and (2) configuring the package as singleton: true in Webpack's shared scope. Webpack then ensures only the highest compatible version is loaded once, and all remotes reuse it.

Writing state (MFE Products)

mfe-products — app.ts
export class App implements OnInit {
  addedIds = signal(new Set<number>());

  constructor(private sharedState: SharedStateService) {}

  ngOnInit() {
    this.sharedState.state$.subscribe(
      s => this.addedIds.set(new Set(s.map(c => c.id)))
    );
  }

  addToCart(course: Course) {
    this.sharedState.addCourse(course);
  }
}

Reading state (MFE Cart)

mfe-cart — app.ts
export class App implements OnInit {
  state = signal<Course[]>([]);
  total = computed(() => this.state().reduce((sum, c) => sum + c.price, 0));

  constructor(private sharedState: SharedStateService) {}

  ngOnInit() {
    this.sharedState.state$.subscribe(s => this.state.set(s));
  }
}

Cart badge in Shell

shell — app.ts
ngOnInit() {
  this.sharedState.state$.subscribe(s => this.cartCount.set(s.length));
}
💡 Why BehaviorSubject? It holds the latest value and immediately emits to new subscribers so a remote that loads late still gets the current state without waiting for the next emission. A plain Subject would miss the value.
05

Deployment — GitHub Actions + GitHub Pages

📚 Theory

Each MFE is deployed independently to its own GitHub Pages site. GitHub Actions triggers a build on every push to master. The Shell's manifest is environment-aware , it points to localhost during development and to the live GitHub Pages URLs in production, driven by Angular's fileReplacements at build time.

GitHub Actions workflow (per MFE app)

.github/workflows/deploy.yml
name: Deploy to GitHub Pages
on:
  push:
    branches: [master]

jobs:
  deploy:
    runs-on: ubuntu-latest
    permissions:
      contents: write
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '20'
          registry-url: 'https://npm.pkg.github.com'
          scope: '@your-org'
      - run: npm install
        env:
          NODE_AUTH_TOKEN: ${{ '{{' }} secrets.PACKAGES_TOKEN {{ '}}' }}
      - run: npm run build -- --base-href /shell/
        env:
          NODE_AUTH_TOKEN: ${{ '{{' }} secrets.PACKAGES_TOKEN {{ '}}' }}
      - run: cp dist/shell/index.html dist/shell/404.html
      - uses: peaceiris/actions-gh-pages@v3
        with:
          github_token: ${{ '{{' }} secrets.GITHUB_TOKEN {{ '}}' }}
          publish_dir: ./dist/shell

Shared library auto-publish workflow

.github/workflows/publish.yml (key steps)
- name: Bump patch version
  run: npm version patch --no-git-tag-version
  working-directory: projects/shared-state

- name: Build library
  run: npx ng build shared-state

- name: Publish
  run: cd dist/shared-state && npm publish
  env:
    NODE_AUTH_TOKEN: ${{ '{{' }} secrets.PACKAGES_TOKEN {{ '}}' }}
💡 Why copy index.html to 404.html? GitHub Pages is a static file server. When a user refreshes on a deep route like /shell/products, GitHub returns a 404. By making 404.html identical to index.html, the Angular app boots and its router handles the URL correctly.
06

Common Pitfalls & Fixes

🚧 "Shared module is not available for eager consumption"

Cause: Angular bootstraps eagerly in main.ts before Webpack can negotiate shared modules.

Fix: Move all code to bootstrap.ts and use a dynamic import in main.ts:

import('./bootstrap').catch(err => console.error(err));
🚧 Singleton service not shared — each MFE has its own instance

Cause: The shared library is not listed in the shared config, or singleton: true is missing.

Fix: Add the library explicitly to every MFE's webpack config:

'@your-org/shared-state': { singleton: true, requiredVersion: 'auto' }
🚧 Backslash in exposes path on Windows

Cause: Windows uses backslashes and some tools auto-generate paths like .\src\app\app.ts.

Fix: Always use forward slashes in webpack.config.js:

exposes: { './Component': './src/app/app.ts' }
🚧 TypeScript rootDir error when using path aliases

Cause: A path alias points outside the project's rootDir, causing TS to reject it.

Fix: Set rootDir to a parent directory in tsconfig.app.json:

"rootDir": ".."
🚧 Cannot GET /assets/module-federation.manifest.json

Cause: Manifest placed in src/assets/ but the path in main.ts doesn't match, or the file isn't in public/.

Fix: Place manifest in public/ and reference it via the environment path. Angular copies public/ to the dist root automatically.

🚧 Routing breaks on page refresh (GitHub Pages)

Cause: GitHub Pages returns 404 for deep routes — there is no file at /shell/products.

Fix: Copy index.html as 404.html in the CI deploy step. GitHub Pages serves 404.html on unknown paths, which boots Angular and lets its router handle the URL.

🚧 Version conflict — strictVersion error at runtime

Cause: Two remotes declare incompatible required versions of the same dependency.

Fix: Use requiredVersion: 'auto' so Webpack reads the version from package.json automatically, or align versions across all apps. As a last resort use strictVersion: false to demote to a warning.

🚧 CORS error fetching remoteEntry.js

Cause: The remote is hosted on a different origin without CORS headers.

Fix: Configure the remote server to return Access-Control-Allow-Origin: * (or the shell's specific origin). GitHub Pages allows cross-origin by default, so this mostly affects custom hosting.

🚧 npm publish 403 — Permission denied

Cause: The GitHub token used in CI doesn't have write:packages scope.

Fix: Create a Personal Access Token (PAT) with write:packages scope, add it as a repository secret, and reference it as NODE_AUTH_TOKEN in the workflow.

07

Interview Q & A

What is Module Federation?
Module Federation is a Webpack 5 feature that allows a JavaScript application to dynamically load code from another application at runtime. It enables micro frontends by letting a Host consume modules exposed by Remotes without bundling them at build time.
What is the difference between a Host and a Remote?
A Host (Shell) consumes remote modules. It controls routing and the top-level layout. A Remote exposes modules (components, routes) via a remoteEntry.js file that the host fetches at runtime. A single app can be both a host and a remote simultaneously.
What is remoteEntry.js and what does it contain?
remoteEntry.js is a small file generated by Webpack for each remote. It acts as a registry — it declares what modules the remote exposes, what shared dependencies it provides, and their versions. The host downloads this file first before loading any remote code. It does not contain the full remote bundle.
Why do we need bootstrap.ts in MFE?
Webpack Module Federation needs to resolve all shared dependencies before any module executes. By wrapping the Angular bootstrap in a dynamic import('./bootstrap') inside main.ts, we give Webpack the chance to negotiate shared modules asynchronously before Angular starts. Without this, you get the "Shared module is not available for eager consumption" error.
What does singleton: true do in the shared config?
It tells Webpack to load only one instance of that package across all micro frontends in the same browser session — even if multiple remotes declare it as a dependency. This is critical for Angular (one zone.js, one injector) and for shared state services (all MFEs share the same BehaviorSubject instance).
What is strictVersion and when would you turn it off?
strictVersion: true causes Webpack to throw a runtime error if the version of a shared dependency doesn't satisfy the required range — preventing silent version mismatches. Turn it off (false) if you want a console warning instead of an error, typically during incremental migration when remotes are temporarily on different versions.
How does cross-MFE state sharing work without a global store like NgRx?
By publishing a shared Angular library with a service using providedIn: 'root' and configuring it as singleton: true in Webpack's shared scope. All remotes inject the same service instance — changes in Products are immediately visible in Shell's cart badge because they share the same BehaviorSubject in memory. No NgRx needed for simple cross-MFE state.
What is the difference between static and dynamic (manifest-based) remote loading?
Static: Remote URLs are declared in webpack.config.js at build time. Changing a remote URL requires rebuilding the host.

Dynamic (manifest-based): The host fetches a JSON manifest at runtime to discover remote URLs. No rebuild needed — just update the JSON file. This is the approach used in this POC via loadManifest().
How do you handle a remote that fails to load?
Use a .catch() on loadRemoteModule() and return a local fallback component. This prevents the entire shell from crashing if one remote is down or not deployed:

loadRemoteModule(...).then(m => m.App).catch(() => FallbackComponent)

The fallback can show a meaningful message, a skeleton UI, or redirect the user gracefully.
Why does Angular routing break on page refresh in GitHub Pages?
GitHub Pages is a static file server. When you refresh on a route like /shell/products, the server looks for a file at that path, finds nothing, and returns a 404. The fix is to copy index.html as 404.html in the CI deploy step — GitHub Pages serves 404.html on any 404, the Angular app boots, and its router handles the path.
What is the purpose of --base-href in the build?
--base-href /shell/ sets the HTML <base href> tag, which tells Angular's router and the browser where to find static assets. On GitHub Pages, apps are served under a sub-path (e.g. /shell/) without this flag, the router breaks and assets 404 because they are requested from the root path.
How do you share a library across separate Angular projects without a monorepo?
Build the library using ng-packagr and publish it to a package registry (npm, GitHub Packages etc.). Each consuming app installs it as a regular npm dependency. For local development, use TypeScript path aliases pointing to the library source. For production, all apps pull the same versioned release from the registry.
How does Angular's environment file substitution work?
In angular.json, under the production configuration, a fileReplacements array maps environment.ts to environment.prod.ts. When ng build runs (defaults to production), every import of environment.ts is transparently replaced with environment.prod.ts no code changes needed.
Can MFEs use different Angular versions?
Technically yes, but it is complex. If versions are too different, the singleton constraint may be violated and Angular will load multiple copies, breaking DI and zone.js. The safest approach is to keep all MFEs on the same major Angular version. For truly different frameworks (e.g. one React, one Angular), Module Federation supports it but you lose singleton Angular benefits.
How do you communicate between two remotes without going through the Shell?
Via the shared singleton service from the shared library — both remotes inject it directly. They do not need to route messages through the shell. Alternative patterns include: a shared event bus (using RxJS Subject in the shared lib), browser Custom Events, or a shared URL/query-param convention for loose coupling.
How do you navigate from inside a remote to a Shell route without a full reload?
Since Angular is a singleton in the shared scope, the Shell's Router instance is available to all remotes. Inject Router in the remote component and call this.router.navigate(['/shell-route']). This triggers Angular's client-side navigation — no page reload. window.location.href must be avoided as it causes a full browser reload.
What are the main challenges of Micro Frontend architecture?
1. Bundle size: Shared dependencies like Angular can be duplicated if misconfigured.
2. Version mismatches: Remotes on different Angular versions may break the shared scope.
3. Routing conflicts: Deep linking across remotes needs careful coordination.
4. Deployment coordination: Breaking changes in a shared library require updating all consumers.
5. Testing complexity: Integration testing across independently deployed apps is harder.
6. Operational overhead: Each MFE needs its own CI/CD pipeline, monitoring, and deploy strategy.
What is requiredVersion: 'auto'?
It tells the Module Federation plugin to automatically read the required version of a shared dependency from the app's package.json at build time so you don't have to hardcode version strings. This reduces maintenance overhead and prevents version drift between what's declared in the config and what's actually installed.
What is shareAll() and should you always use it?
shareAll() is a helper that automatically shares every dependency listed in package.json as a singleton. It is convenient for getting started but shares more than necessary. In production, prefer explicitly listing only the packages that need to be shared (Angular, RxJS, your shared lib) to keep the shared scope clean and predictable.
